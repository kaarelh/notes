

# do bayes-NNs have the kolmogorov prior? (my answer: no)

I spent a few hours thinking about the Dingle, Valle-Pérez, Mingard, et al "the bayesian NN prior is sorta given by kolmogorov complexity" literature (e.g. https://arxiv.org/pdf/1805.08522 and https://www.nature.com/articles/s41467-018-03101-6 ). I think I maybe now understand a bit better where Lucius was coming from with the claims that the architecture having some kind of degeneracy is enough to imply having good generalization behavior (though I still think the claims are wrong). I think I probably have a decent picture of what's going on in the vicinity of this literature now. The point of this message is to briefly communicate the state of things here as I see it (you might be familiar with some of this already).

First, it's true (and sorta interesting) that complicated functions must get low probability:
Proposition (informal version). Let $p(f)$ denote the probability given to a boolean function $f\colon \{-1,1\}^n\to\{-1,1\}$ by an architecture+prior and $K(f)$ denote its kolmogorov complexity[^1]. Then $p(f)\leq 2^{-K(f)}$ up to some plausibly-smaller terms for reasonable architectures+priors.
Proposition (formal version). We have $p(f)/(log(1/p(f)))\leq 2^{-K(f)+c+O(1)}$, where $c$ is the complexity of specifying the architecture and the prior[^2] and $O(1)$ is literally some constant that doesn't even depend on the architecture which we could state explicitly if we wanted.
Pf. We can write a program which prints $f$ consisting of the following routines:
1) specify the architecture and the prior (this takes $c$ bits);
2) for every function, compute how much prior the architecture+prior gives it (just by looping through all param settings);
3) delete all functions with probability strictly below $p(f)$ (let's round down to the nearest $2^{-i}$, actually. so specifying this cutoff takes at most about $\log(i) = \log(\log(1/p(f)))$ bits[^3]);
4) specify a pointer into the remaining list to say which one $f$ is (the list has at most $1/p(f)$ elements because probabilities of remaining elements must sum to at most $1$, so this takes at most $\log(1/p(f))$ bits);
5) print $f$.
This is a way to specify $f$ with $\log(1/p(f))+\log(\log(1/p(f)))+c+O(1)$ bits, so $K(f) \leq \log(1/p(f))+\log(\log(1/p(f)))+c+O(1)$, so $2^{-K(f)+c+O(1)}\geq p(f)/(\log(1/p(f)))$, which is what we wanted to show.

This proposition says that if a function gets high prior, it needs to have low kolmogorov complexity. It in particular says that complicated (i.e. typical) functions, i.e. those with $K(f)\approx 2^n$, will get probability roughly at most $2^{-2^n}$ — i.e., what they'd be given by the uniform prior on functions. However, we haven't provided any argument for simple functions getting higher probability than that either, and this is in fact false for general "architectures", as one can e.g. easily get the uniform prior on functions, which always thinks any unseen output is equally likely to be $-1$ or $1$, or one can have an architecture which always predicts that any unseen output is $1$, or that any unseen output is $-1$, or many other stupid things[^4]. Even if you include some degeneracy assumption of the form "at least some functions should get pretty high probability", I think this will only let you conclude that some simple functions get high probability, not that all simple functions get high probability or that simple functions generally get high probability, which I feel is what you'd want to have to conclude that there's good generalization behavior. So, I doubt there is anything that interesting to be said on this level of generality even if you have some degeneracy assumption.[^5] Maybe a central broad obstacle to there being any interesting lower bound saying that kolmogorov-simple functions must be given decent probability is that there just are functions with simple programs but no small circuits, and eg an MLP is a circuity thing so you won't get a high prior on those functions, so there couldn't be any general matching lower bounds on the MLP prior.[^6]

Anyway, I still think that to get a decent explanation of NN-bayes generalizing well[^7], one should probably make more contact with the architecture+prior (given the task), because one simply doesn't get good generalization behavior in this generality, and I think one should probably be attempting to build a bridge to some more circuity thing (which still feels principled — eg a boolean circuit simplicity prior for boolean functions), because one probably just doesn't get program-like generalization from NNs (which are a circuity thing); I outline a highly incomplete proposal for this in my comment: https://www.lesswrong.com/posts/LxCeyxH3fBSmd4oWB/deep-learning-is-cheap-solomonoff-induction-1?commentId=vPJfA4G6wzC5SPeXq . Alternatively, this could look like talking about which kinds of circuits are $L^2$-cheap. And to get a decent explanation of SGD-NNs generalizing well[^8], my best bet would be to try to understand the local building process which is going on, e.g. talking about which circuit components are cheap to build and how this makes you build something good.

this does let us make the following plausibly interesting statement about any bayesNN prior: the prior is given by picking some subset of all the functions and roughly giving the kolmogorov prior probability to each of those (up to smaller terms and plus another prior of total measure o(1) on the complement of those functions)

# some random sass about this literature
* i feel like it's pretty badvibes/confused to call eg lempel-ziv complexity a "computable approximation of kolmogorov complexity". it's like calling proving a single theorem a computable approximation of having solved math or something. like to spell out the point more: lempel-ziv is using one idea for compression, whereas kolmogorov complexity is using, like, all the ideas

[^1]: you can think of $f$ as just a string listing the outputs on the inputs in the canonical order on inputs and ask for the kolmogorov complexity of this string; alternatively, you can equivalently-up-to-lower-terms just ask for the size of the smallest program computing $f$
[^2]: Note in particular that $c$ doesn't depend on $f$, and it's probably fine to think of it as typically being like $\log n$, whereas $K(f)$ is between $0$ and (roughly) $2^n$.
[^3]: can one be more clever here, choosing a lower-complexity cutoff without being too loose with the bound (being loose is bad because it makes the list we have to specify a pointer into bigger)? i think there is probably a counting argument showing one typically can't deal with this tradeoff significantly better, but i haven't thought this through carefully, and idk maybe there's some other way to glue things in this argument which avoids the $\log$ term we're picking up here
[^4]: The example I'm about to give in this footnote will be stretching the notion of "architecture", but if all one means is any low-complexity rule for going from a "parameter choice set" $\{1,2,\ldots,m\}$ to functions, one could even make a loop through param settings for the architecture be like a loop over all functions, except with a very big (but bounded by some low-complexity number, e.g., size ackermann function of $(n,n)$) search for short fast programs computing the function, rejecting and passing over every function for which a short fast program is found; this gives some prior supported only on 'complicated functions', i.e., with 0 probability assigned to any simple fast function!
[^5]: Well ok, you could of course do it if you have a degeneracy assumption which says that each function has probability as big as it could have given this bound, but that feels very close to begging the question.
